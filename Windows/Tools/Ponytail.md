# 使用Ponytail

## 写新功能

```
@ponytail 给项目增加 xxx 功能。
要求：
1. 先检查是否已有标准库/原生能力/已有依赖可以做；
2. 不新增依赖，除非必须；
3. 不增加抽象层；
4. 改动文件越少越好；
5. 非平凡逻辑加一个最小可运行测试；
6. 如果用了简化方案，用 ponytail: 注释写明限制和后续升级路径。
```

## 审查已有代码

```
@ponytail-review 审查当前 diff。
输出：
1. 可以直接删除的代码；
2. 可以用标准库替代的代码；
3. 不必要的抽象/封装；
4. 风险点：安全、输入校验、数据丢失、可访问性不要误删；
5. 最小修改补丁建议。
```

## 优化整个项目

```
@ponytail-audit 扫描整个仓库，找过度工程化问题。
请按收益排序：
1. 删除无用代码；
2. 合并重复逻辑；
3. 移除不必要依赖；
4. 简化配置；
5. 替换为标准库或平台能力；
6. 给出每项修改的风险和测试方式。
```
