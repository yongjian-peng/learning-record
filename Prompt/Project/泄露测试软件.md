## 2. Heob 友好化改造

目标：方便使用 Heob 检测内存泄漏、越界访问、线程退出问题、对象未释放问题。

请重点检查：

1. 是否存在裸指针 `new` 后没有 parent，也没有 delete 的情况。

   * 如果是 QObject，优先设置 parent。
   * 如果不是 QObject，优先改成：

     ```cpp
     std::unique_ptr
     ```

     或者合适的 RAII 管理方式。

2. 检查 QThread 使用是否安全：

   * 退出时是否调用 `quit()`
   * 是否调用 `wait()`
   * worker 是否正确 deleteLater
   * 是否存在程序退出后线程还活着的问题

3. 检查串口、Modbus、PLC、数据库连接、定时器、曲线数据缓存是否正确释放。

4. 检查析构函数中是否应该补充：

   * stop service
   * disconnect
   * stop timer
   * close database
   * quit/wait thread

5. 对测试曲线数据、报警列表、实时数据缓存做内存上限保护。

   * 例如曲线点数不要无限增长。
   * 可以设置最大缓存数量，比如 3000 或 10000，根据当前业务代码合理判断。

6. 检查 C++ 和 QML 交互中是否有对象所有权问题。

   * 通过 `setContextProperty` 暴露给 QML 的对象，必须由 C++ 明确管理生命周期。
   * 动态创建的 QObject 不要让 QML 和 C++ 双方都误以为自己拥有对象。

7. 不要为了消除 Heob 报告而隐藏问题，不要强行忽略真实泄漏。请真正修复代码中的生命周期问题。

需求拆解
PPT 里的核心业务可以归纳为：
设备输入：环境温度，初级表 6 路压力，次级表 2 路压力，规格包含 0-5V / 4-20mA / CANopen
外设：PLC、阀门、触摸屏、工控机、显示器、打印机
管理功能：登录、管理员参数设置、配方新建/查询/更新
测试流程：选择配方，录入物料号/工单号，确认安装，开始测试，PLC 开阀，2 秒采样，曲线显示

界面层建议
QSkinny 本身已经把控件、皮肤、渲染拆得比较干净，比如 QskControl 管控件行为，QskSkinlet 管渲染节点。业务软件应沿用这个思路：

HomePage：配方选择、物料号、工单号、开始按钮、压力未释放报警
RecipePage：管理员权限，新建/查询/更新配方
TestPage：实时曲线、环境温度、输入压力、压差、实际测试时间、继续/停止/打印序列号

数据模型
建议先固定这些核心表/对象：

Recipe：压力表型号、产品类型、输入压力、泄漏压力限值、PRV 设定值、PRV 开启值、PRV 关闭值、测试时间、序列号格式
WorkOrder：物料号、工单号、操作员、产品类型
TestSession：开始时间、结束时间、当前状态、配方快照、工单信息
PressureSample：时间戳、环境温度、输入压力、各通道压力、压差
TestResult：合格/不合格/中止、失败原因、退出原因、打印状态、导出状态
配方要做“快照保存”，不要只保存 recipe_id。否则以后配方被改了，历史测试记录会失真。

中型项目落地顺序
建议分 4 步走：

先建 LeakTester 独立应用目标，复用 QSkinny，不修改 src 核心库。
先实现 domain + application + mock infrastructure，把完整测试流程跑通。
再接 SQLite、Excel/PDF、打印机。
最后接 PLC/CANopen/传感器，并补设备异常、断线重连、压力未释放报警。
关键建议
这个项目最容易失控的地方不是 UI，而是“测试流程 + 硬件 IO + 数据保存”混在一起。中型项目的边界应该是：

QSkinny 只负责显示和交互
application 负责流程编排
domain 负责判定规则
infrastructure 负责真实设备、数据库、导出、打印
状态机是测试流程的唯一入口
下一步最值得做的是先搭一个 app/leaktester 骨架：CMake 目标、领域模型、测试状态机、Mock PLC/传感器，以及一个最小 QSkinny 主界面。这样架构就能从纸面落到可运行代码上。
